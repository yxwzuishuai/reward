# 订单号生成服务

## 订单号格式（22位）

```
PP + YYYYMMDD + UUUU + SSSSSS + RR
```

| 段 | 位数 | 示例 | 含义 |
|---|---|---|---|
| PP | 2 | OD | 业务前缀，区分单据类型 |
| YYYYMMDD | 8 | 20260319 | 日期，可读，方便按日归档 |
| UUUU | 4 | 0056 | 用户基因，userId % 10000，分库分表路由用 |
| SSSSSS | 6 | 010100 | 机器ID(2位) + 序列号(4位) |
| RR | 2 | 42 | 随机数，防遍历猜测 |

完整示例：`OD20260319005601010042`

## 业务前缀定义

| 前缀 | 含义 |
|---|---|
| OD | 普通订单 |
| RF | 退款单 |
| PT | 拼团单 |
| SK | 秒杀单 |

可按业务需要自行扩展，只要保持2位即可。

## 设计要点

### 1. 唯一性
- 毫秒时间戳 + 机器ID + 自增序列 + 随机数，多维度组合
- 同毫秒内通过 AtomicInteger CAS 自增，支持每毫秒 9999 个订单
- 不同机器通过 machineId（0-99）区分，最多支持 100 个实例

### 2. 可读性
- 人眼可直接识别日期和业务类型
- 客服看到 `OD20260319...` 就知道是3月19日的普通订单
- 方便按日期范围查询和归档

### 3. 基因性（分库分表路由）
- 订单号第10-13位嵌入了 `userId % 10000` 的值
- 分库分表时，无需回查用户表，直接从订单号提取路由键

```java
// 提取路由基因
int gene = OrderNoGenerator.extractUserGene("OD20260319005601010042");
// gene = 56，对应 userId % 10000 = 56
// 路由到第 56 % 分表数 的表
```

### 4. 高性能
- 纯内存计算，无 IO 操作
- 序列号使用 AtomicInteger CAS 操作，无锁竞争
- 不依赖 Redis、数据库或任何外部服务
- 单机 QPS 可达百万级

### 5. 高可用
- 零外部依赖，服务启动即可用
- 不存在 Redis 挂了导致无法生成订单号的问题
- 每个服务实例独立生成，互不影响

### 6. 可扩展性
- 业务前缀支持多种单据类型
- machineId 支持 0-99，满足中等规模集群
- 如需更大规模，可将 machineId 扩展到3位

## 使用方式

```java
// 生成普通订单号
String orderNo = OrderNoGenerator.generate(userId);

// 生成退款单号
String refundNo = OrderNoGenerator.generate("RF", userId);

// 从订单号提取用户基因（分库分表路由）
int gene = OrderNoGenerator.extractUserGene(orderNo);

// 从订单号提取日期
String date = OrderNoGenerator.extractDate(orderNo);
```

## 多实例部署

每台机器启动时设置不同的 machineId，避免序列号冲突：

```java
// 方式1：通过环境变量
OrderNoGenerator.setMachineId(Integer.parseInt(System.getenv("MACHINE_ID")));

// 方式2：通过配置文件
@Value("${app.machine-id:0}")
private int machineId;

@PostConstruct
public void init() {
    OrderNoGenerator.setMachineId(machineId);
}
```

## 并发安全分析

```
线程A ──> System.currentTimeMillis() = 1710835200000
          lastTimestamp = 1710835200000
          SEQUENCE.getAndIncrement() = 0  ──> seq = 0

线程B ──> System.currentTimeMillis() = 1710835200000（同毫秒）
          lastTimestamp 已经是当前值，不重置
          SEQUENCE.getAndIncrement() = 1  ──> seq = 1（CAS保证不重复）

线程C ──> System.currentTimeMillis() = 1710835200001（下一毫秒）
          lastTimestamp 更新，SEQUENCE 重置为 0
          SEQUENCE.getAndIncrement() = 0  ──> seq = 0
```

- `lastTimestamp` 用 volatile 保证可见性
- `SEQUENCE` 用 AtomicInteger 保证原子自增
- 极端情况下同毫秒序列号溢出（>9999），会导致序列号位数变化，但概率极低（需要单毫秒内同一台机器生成1万+订单）
