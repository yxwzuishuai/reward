# 游标分页（Cursor-based Pagination）

## 整体流程

```
客户端请求 → Controller → Service → Mapper(SQL) → 返回结果 + 下一页游标
```

前端每次请求带上上一页返回的 `nextCursor`（上一页最后一条记录的 ID），后端用 `WHERE id > cursor` 直接定位，不用 OFFSET。

## 1. 请求层 — CursorPageRequest

```java
private Long cursor;      // 上一页最后一条的ID，首次传 null 或 0
private Integer pageSize;  // 每页大小，默认20，最大100
```

两个防御方法：
- `getSafeCursor()` 处理 null/负数，返回 0
- `getSafePageSize()` 限制上限为 100，防止一次查太多

## 2. 响应层 — CursorPageResult

```java
public static <T> CursorPageResult<T> of(List<T> records, int pageSize,
                                          Function<T, Long> idExtractor) {
    // 查出的条数 == pageSize → 可能还有下一页
    result.setHasMore(records.size() >= pageSize);
    // 取最后一条的ID作为下一页游标
    result.setNextCursor(idExtractor.apply(records.get(records.size() - 1)));
}
```

关键逻辑：如果查出来的数据刚好等于 pageSize，说明后面可能还有数据，`hasMore = true`，并把最后一条的 ID 作为 `nextCursor` 返回给前端。

## 3. SQL 核心 — 延迟关联 + 游标

这是最精华的部分。Mapper 里的 SQL 用了「延迟关联（Deferred Join）」技巧：

```sql
SELECT o.*, u.user_name, p.product_name, s.shop_name
FROM orders o
INNER JOIN (
    -- 第一步：子查询只查ID，走覆盖索引，极快
    SELECT id FROM orders WHERE id > ? ORDER BY id ASC LIMIT ?
) tmp ON o.id = tmp.id
-- 第二步：只对这几条记录做多表join
LEFT JOIN users u ON o.user_id = u.id
LEFT JOIN products p ON o.product_id = p.id
LEFT JOIN shops s ON o.shop_id = s.id
```

### 为什么用延迟关联？

- 内层子查询 `SELECT id FROM orders WHERE id > ? LIMIT ?` 只查主键，走覆盖索引（只扫 B+ 树索引，不回表读数据行），3000 万数据也是毫秒级
- 外层拿到这几十个 ID 后再做多表 join，回表次数就是 pageSize 次，可控且很快
- 如果直接 `SELECT * ... JOIN ... LIMIT`，MySQL 可能先做全量 join 再截取，代价大得多

## 4. 带条件查询

当有 `status` 筛选时，条件加在子查询里面：

```sql
SELECT id FROM orders WHERE id > ? AND status = ? ORDER BY id ASC LIMIT ?
```

这里能走 `idx_status_id(status, id)` 联合索引 — 等值条件 `status` 在前，范围条件 `id` 在后，索引设计和查询完美匹配。

## 5. 使用示例

```
第一页: GET /order/page?pageSize=20
下一页: GET /order/page?cursor=10020&pageSize=20   (cursor 为上一页返回的 nextCursor)
带条件: GET /order/page?cursor=10020&pageSize=20&status=1
```

## 6. 传统分页 vs 游标分页

| | 传统 OFFSET | 游标分页 |
|---|---|---|
| 第 1 页 | ~5ms | ~3ms |
| 第 50 万页 | ~30s | ~3ms |
| 原理 | 扫描并丢弃前 N 行 | B+ 树直接定位 |
| 缺点 | 越翻越慢 | 不能跳页，只能上/下翻 |

## 7. 适用场景

游标分页的代价是不支持"跳到第 N 页"，只能顺序翻页，适合以下场景：
- 无限滚动 / 瀑布流
- 移动端列表加载更多
- 数据量大（百万级以上）的后台分页
- 对翻页性能有严格要求的接口


## 8. 混合分页（Hybrid Pagination）

纯游标分页不支持跳页，后台管理系统往往需要页码跳转。混合分页的策略：前 500 页用传统 OFFSET（支持跳页），超过阈值自动切换为游标分页（保证性能）。

接口示例：
```
跳页:   GET /order/hybrid-page?pageNum=100&pageSize=20
带条件: GET /order/hybrid-page?pageNum=100&pageSize=20&status=1
深翻页: GET /order/hybrid-page?cursor=200000&pageSize=20  (自动走游标)
```

响应中的 `mode` 字段会告诉前端当前使用的是 `offset` 还是 `cursor` 模式。OFFSET 模式下额外返回 `total` 和 `pageNum`，方便前端渲染页码。

## 9. 泛型设计解析

### 类上的 `<T>` — 让一个类适配所有数据类型

```java
public class HybridPageResult<T> {
    private List<T> records;   // T 是什么，records 就装什么
}
```

用的时候，创建对象时决定 T 的具体类型：

```java
HybridPageResult<OrderDetailDTO> orderPage = new HybridPageResult<>();  // T = OrderDetailDTO
HybridPageResult<UserDTO> userPage = new HybridPageResult<>();          // T = UserDTO
```

如果不加 `<T>`，就得为每种数据写一个分页类，泛型让一个类搞定所有类型。

### static 方法上的 `<T>` — 自己声明泛型

```java
public static <T> HybridPageResult<T> ofCursor(List<T> records, int pageSize, Function<T, Long> idExtractor)
//             ^^^
//             这个 <T> 是方法级泛型声明
```

为什么 static 方法要单独声明 `<T>`？

- 类上的 `T` 是在创建对象（`new`）时确定的
- static 方法通过类名直接调用（`HybridPageResult.ofCursor(...)`），没有对象，类上的 `T` 还没被赋值
- 所以 static 方法必须自己声明 `<T>`，编译器会从传入的参数自动推断 T 的具体类型

```java
// 传入 List<OrderDetailDTO>，编译器推断 T = OrderDetailDTO
HybridPageResult.ofCursor(orderList, 20, OrderDetailDTO::getOrderId);

// 传入 List<UserDTO>，编译器推断 T = UserDTO
HybridPageResult.ofCursor(userList, 20, UserDTO::getUserId);
```

简单说：第一个 `<T>` 是"声明"，后面的 `HybridPageResult<T>`、`List<T>`、`Function<T, Long>` 是"使用"。就像变量要先定义才能用一样。
